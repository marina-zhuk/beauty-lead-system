# Beauty Lead System

## Назначение проекта

Это коммерческий MVP для портфолио: мини-система заявок для салона красоты или частного beauty-мастера.

Система должна включать:
- лендинг;
- форму заявки;
- обработку заявки через API route;
- Telegram-уведомление владельцу;
- адаптивный UI;
- README с инструкцией запуска.

## Целевая аудитория

Проект предназначен для:
- мастеров маникюра;
- бровистов;
- lash-мастеров;
- косметологов;
- массажистов;
- небольших салонов красоты.

## Главная бизнес-цель

Показать потенциальному клиенту, что система помогает не терять заявки:
клиент оставляет заявку → владелец получает уведомление → заявка сохраняется в таблицу.

## Stack

Использовать:
- Next.js App Router;
- TypeScript;
- Tailwind CSS;
- shadcn/ui по необходимости;
- Zod для валидации;
- Next.js API route;
- Telegram Bot API;
- Vercel для деплоя.

## Основной пользовательский сценарий

1. Клиент открывает лендинг.
2. Смотрит услуги.
3. Нажимает кнопку "Записаться".
4. Заполняет форму:
   - имя;
   - телефон;
   - услуга;
   - желаемая дата;
   - желаемое время;
   - комментарий.
5. Отправляет заявку.
6. Видит сообщение об успешной отправке.
7. Владелец получает Telegram-уведомление.

## Поля заявки

- createdAt
- name
- phone
- service
- preferredDate
- preferredTime
- comment
- source
- status

## Статусы заявки

- new
- contacted
- booked
- cancelled
- completed

По умолчанию новая заявка получает статус `new`.

## Требования к UI

Интерфейс должен быть:
- чистым;
- современным;
- адаптивным;
- понятным на телефоне;
- без визуального шума;
- без перегруженных анимаций.

Стиль:
- светлый фон;
- аккуратные карточки;
- понятные CTA-кнопки;
- секции с нормальными отступами;
- mobile-first подход.

## Структура лендинга

Главная страница должна включать:

1. Hero
   - оффер;
   - короткое объяснение;
   - CTA-кнопка "Записаться";
   - вторичная кнопка "Посмотреть услуги".

2. Services
   - 4–6 услуг;
   - название;
   - короткое описание;
   - цена "от ... ₽".

3. Benefits
   - почему удобно записываться через систему;
   - почему владелец не теряет заявки.

4. Process
   - 3 шага:
     1. клиент оставляет заявку;
     2. владелец получает уведомление;
     3. заявка сохраняется в таблицу.

5. LeadForm
   - форма заявки.

6. Reviews
   - 2–3 демо-отзыва, явно как примеры.

7. FAQ
   - частые вопросы.

8. Footer
   - контакты;
   - подпись demo project.

## Требования к форме

Форма должна иметь поля:
- name;
- phone;
- service;
- preferredDate;
- preferredTime;
- comment.

Форма должна иметь состояния:
- idle;
- loading;
- success;
- error.

Валидация:
- name обязательно;
- phone обязательно;
- service обязательно;
- preferredDate желательно;
- preferredTime желательно;
- comment опционально.

## Backend

Создать API route:

POST /api/lead

Он должен:
1. принять данные формы;
2. валидировать данные через Zod;
3. вернуть ошибку, если данные некорректные;
4. на первом этапе просто вернуть success;
5. на следующих этапах отправлять Telegram.

## Безопасность

- Не хранить токены в коде.
- Использовать `.env.local`.
- Добавить `.env.local.example`.
- Не коммитить `.env.local`.
- Не логировать Telegram token.
- Не показывать пользователю технические ошибки.
- Не добавлять секреты в README.

## Ограничения MVP

В первой версии не добавлять:
- оплату;
- авторизацию;
- личный кабинет;
- полноценную CRM;
- Telegram Mini App;
- AI-chatbot;
- сложную аналитику;
- базу данных;
- лишние библиотеки.

## Definition of Done

Проект считается готовым, если:

- сайт запускается локально;
- главная страница отображается без ошибок;
- форма работает;
- данные формы валидируются;
- пустая форма не отправляется;
- заявка уходит в Telegram;
- пользователь видит success/error состояние;
- есть `.env.local.example`;
- есть README;
- проект можно задеплоить на Vercel;
- интерфейс нормально выглядит на телефоне.
